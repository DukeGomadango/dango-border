特徴量定義（確定版 / C-1実装基準）

1) カレンダー基本特徴量（全ターゲット共通）
- date: YYYY-MM-DD
- year, month, day
- weekday_num (0=月 ... 6=日)
- weekday_text（元データの曜日文字）
- quarter
- is_month_start, is_month_end
- season（spring/summer/autumn/winter）

2) 休日・イベント特徴量
- is_holiday_jp（日本祝日フラグ）
- is_day_before_holiday
- is_day_after_holiday
- is_long_weekend_window（3連休前後）
※MVPは固定カレンダーファイルで管理。外部API依存は持たない。

3) ターゲット時系列特徴量（ターゲットごと）
- lag_1, lag_7, lag_14, lag_28
- rolling_mean_7, rolling_mean_28
- rolling_std_7, rolling_std_28
- weekday_seasonal_mean（同曜日の過去平均）
※未来情報を使わないよう時系列順に生成する。

4) 欠損値ルール
- 元データの「未登録」は欠損（NaN）として保持
- 学習時の補完はモデル設定で分離（前処理で0埋めしない）
- 補完戦略はターゲット別に管理し、実験ログへ記録

5) 採用ポリシー（ターゲット別）
- Step1: カレンダー基本 + lagのみ
- Step2: rolling/seasonal追加
- Step3: 休日特徴量追加
- 各Stepで時系列CVを行い、ベースライン改善時のみ採用

6) リーク防止・再現性
- splitはrolling/expanding window固定
- 特徴量生成コードのバージョンを保存
- 学習seed固定
- データスナップショット（入力ファイルID）をモデルに紐付け

月次運用手順（自分運用 / 方針A）

前提: 生データは `raw-data.xlsx` をプロジェクト直下に置く。人手での Excel 加工はしない。

1) API 起動（常駐ターミナル）

```powershell
cd c:\Users\furup\Documents\border-analysis
.\scripts\start-api.ps1
```

起動ログに `Application startup complete` が出ること。依存は起動時に `requirements.txt` から自動インストール。

2) データ投入（新しい月次ファイルがあるとき）

```powershell
.\scripts\verify-m1.ps1
```

または UI / `POST /datasets/upload` で `raw-data.xlsx` をアップロード。品質ゲートで `pipeline_blocked` になった場合は理由を確認し、必要なら override 後に学習へ進む。

3) 公開ターゲット一括再学習（strategy=auto）

```powershell
.\scripts\train-ready.ps1
```

`adopted` が active モデル数。`trained but not adopted` が出たら手順 3b へ。

3b) 未採用ターゲット（auto_steps で 10% 改善に届かない列）

```powershell
.\scripts\retry-not-adopted.ps1
```

- 通常学習は `feature_profile=auto_steps`（step1→step3 を CV、最良を採用）。
- **1回だけ** `sparse`（lag 短縮・祝日なし）で再評価する（retry スクリプト）。
- 採用されたら active が切り替わる。まだ未採用なら **直前の active を維持**（推論は継続可能）。
- `compare-strategies` で linear / lightgbm / auto も全部未採用なら、無理に閾値を下げない（lag-7 ベースラインが既に強い列）。

例（2026-06 時点）: `C5 +2`, `C5 +4` は sparse で採用可。`C3 +2` は full/sparse とも未採用のため旧モデル維持。

4) 状態確認

```powershell
.\scripts\report-targets.ps1
```

`publish=Y` かつ `active=Y` を確認。推論日は `predict_to` 以内（UI の予測可能範囲と一致）。

5) スポット推論

- UI: http://127.0.0.1:8000/ui/
- API 例: `GET /predictions?target=S1%20%2B6&date=2026-02-26`（日付はターゲットごとの `predict_to` を参照）

6) 戦略比較（任意・精度見直し時）

```powershell
.\scripts\compare-strategies.ps1
# ターゲット指定例:
.\scripts\compare-strategies.ps1 -Target "B1 +6","C3 +2"
```

linear / lightgbm / auto の CV MAE を比較。採用基準はベースライン比 10% 以上改善。デフォルトは `S1 +6` と要監視列 `C3 +2`。

7) M2 機能（実装済み）

- 期間予測: `GET /predictions/range`、予測 UI の「期間予測」ボタン
- ターゲット運用 UI: http://127.0.0.1:8000/ui/targets.html（公開切替・beta 表示・学習状態）
- メトリクス: `GET /system/metrics`、監査 `GET /system/audit`
- 予測精度ログ: 推論時に `storage/monitoring/predictions.jsonl`、取込後に自動 reconcile

8) M3 段階公開（beta → 合計39公開）

```powershell
# 1) beta 全件を学習（publish 不要）
.\scripts\train-beta.ps1

# 2) 公開計画と候補を確認
curl.exe "http://127.0.0.1:8000/datasets/publication/plan"
curl.exe "http://127.0.0.1:8000/datasets/publication/candidates"

# 3) adopted かつ active な beta を残りスロットまで一括公開
.\scripts\promote-beta.ps1
```

- 目標: 公開 `publish=true` が **39**（ready 30 + beta 最大9）
- 公開条件: `status=beta` + active モデル + `evaluation.adopted=true`
- UI: http://127.0.0.1:8000/ui/targets.html（M3 セクション）
